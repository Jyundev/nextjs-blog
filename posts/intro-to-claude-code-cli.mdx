요즘 AI 도구를 쓰는 흐름이 정해져 있다.

에디터에서 코드를 복사하고, 브라우저 탭을 열어 붙여넣고, 답변이 오면 다시 에디터로 가져온다. 도구가 좋아질수록 이 루프가 빨라질 뿐, 근본적인 구조는 바뀌지 않았다.

Claude Code는 그 구조 자체가 다르다. 터미널 안에서 직접 파일을 읽고, 명령어를 실행하고, 코드를 수정한다. 처음 실행했을 때 이런 메시지가 떴다.

<br></br>

```
╭─────────────────────────────────────────────╮
│ ✻ Welcome to Claude Code!                   │
│                                             │
│   /help for help, /status for your plan     │
╰─────────────────────────────────────────────╯

 Quick safety check: Do you trust the files in this folder?
```

<br></br>

"이 폴더의 파일을 신뢰하냐"는 질문이었다. 처음엔 그냥 넘겼지만, 왜 이 질문이 필요한지는 좀 더 쓰다 보니 이해가 됐다.

---

## Claude Code란

Claude Code는 Anthropic이 만든 터미널 기반 AI 에이전트다.

단순히 코드를 생성해주는 것이 아니라, 실제로 로컬 파일 시스템에 접근해서 읽고, 수정하고, 터미널 명령어를 실행하며, Git 커밋까지 제안한다. 설치는 간단하다.

<br></br>

```bash
npm install -g @anthropic-ai/claude-code
```

<br></br>

프로젝트 루트에서 `claude`를 실행하면 대화형 세션이 시작된다.

---

## `/init` — 프로젝트 구조를 스스로 파악한다

Claude Code에서 가장 인상적인 기능은 `/init`이다.

이 명령어를 실행하면 Claude가 `package.json`, `tsconfig.json`, 라우트 구조, 유틸리티 파일들을 직접 열어 읽기 시작한다. 몇 분 뒤 `CLAUDE.md` 파일이 생성된다.

이 블로그 프로젝트를 대상으로 실행해봤을 때, Claude가 스스로 파악해서 기록한 내용들은 다음과 같았다.

- `pnpm dev`는 단순히 개발 서버를 여는 명령어가 아니라, 먼저 `generate-mdx-components` 스크립트를 실행한다는 것
- `app/components/mdxComponents/index.ts`는 자동 생성 파일이므로 직접 편집해선 안 된다는 것
- 새 MDX 컴포넌트를 추가하려면 `.tsx` 파일만 만들면 되고, 다음 `pnpm dev` 실행 시 자동으로 등록된다는 것

<br></br>

```json
"dev": "pnpm run generate-mdx-components && next dev"
```

<br></br>

`package.json` 한 줄에서 `dev` 스크립트가 두 단계로 구성되어 있다는 맥락까지 읽어낸 것이다. 내가 직접 온보딩 문서를 작성했다면 놓쳤을 법한 내용들이었다.

생성된 `CLAUDE.md`는 이후 세션에서 자동으로 컨텍스트에 포함된다. 다음에 Claude Code를 열 때는 "이 프로젝트가 어떻게 동작하는지"를 다시 설명할 필요가 없다.

---

## 실전에서 확인한 것들

이 블로그에서 새 MDX 컴포넌트를 추가하는 상황을 예로 들면, 기존에는 다음 순서를 직접 밟아야 했다.

1. `app/components/mdxComponents/` 하위에 `.tsx` 파일 생성
2. `pnpm run generate-mdx-components` 실행해서 `index.ts` 갱신
3. MDX 파일에서 컴포넌트 사용

<br></br>

이 흐름에서 2번 단계를 빠뜨리거나, `index.ts`를 직접 편집하려 하면 문제가 생긴다. 특히 `index.ts`는 자동 생성 파일이라 수동으로 건드리면 다음 번 스크립트 실행 시 덮어씌워진다.

Claude Code는 `CLAUDE.md`에 기록된 이 규칙을 알고 있기 때문에, 컴포넌트 추가를 요청하면 이 흐름을 그대로 따른다. 실수가 발생하는 지점을 미리 이해하고 있는 것이다.

## Quick safety check — 신뢰는 명시적으로

처음 실행 시 떴던 그 질문으로 돌아가 보자.

Claude Code는 파일을 읽고, 수정하고, 터미널 명령을 실행하는 권한을 가진다. 만약 악의적인 내용이 담긴 외부 저장소를 클론한 뒤 무심코 Claude Code를 실행하면, `CLAUDE.md`나 `.claude/` 설정 파일에 의도하지 않은 지시가 들어있을 수 있다.

신뢰 확인은 "이 프로젝트의 코드와 설정을 Claude가 실행해도 되는지"를 개발자가 명시적으로 승인하는 단계다. 그 외에도 파일 수정이나 터미널 명령 실행 전에 Claude는 항상 승인을 구한다. 귀찮은 절차가 아니라, 이 도구가 가진 권한에 비례하는 경계선이다.

---

## 이 글도 Claude Code와 함께 썼다

이 글은 Claude Code CLI를 소개하는 내용이지만, 실제로 이 블로그 포스트 자체도 Claude Code와 함께 작성했다.

MDX 파일 생성부터 프로젝트의 `<Table>` 컴포넌트 형식 확인, `<br></br>` 간격 패턴 파악까지 Claude가 기존 코드를 직접 읽고 맞춰줬다. 설명하지 않아도 이 프로젝트의 규칙을 알고 있었기 때문이다.

---

## 정리

Claude Code를 쓰면서 가장 달라진 점은, 도구에게 컨텍스트를 전달하는 방식이다.

기존 AI 도구들은 매번 "이 프로젝트는 이런 구조입니다"를 설명해야 했다. Claude Code는 코드베이스를 직접 읽고 그 맥락을 스스로 쌓는다. 덕분에 대화의 출발점이 달라진다.

단순한 코드 생성기라기보다, 프로젝트를 이해하고 있는 동료가 터미널에 상주하는 느낌에 가깝다.

아직 완벽하진 않다. 가끔 의도를 잘못 해석하거나 예상치 못한 파일에 손을 대려 할 때도 있다. 그래서 모든 실행 전에 승인을 묻는 구조는 지금 단계에서 맞는 설계라고 생각한다.

어디까지 발전할지는 모르겠지만, 지금 이 시점에서도 충분히 실용적이다.
